## 1. 들어가기 전에

Claude Code와 대화할 때, 우리가 사용할 수 있는 도구가 세 가지 있습니다.

| 도구 | 비유 | 한 줄 설명 |
|------|------|-----------|
| **커맨드** | 레시피 카드 📋 | "이렇게 해줘"라고 미리 적어둔 지시서 |
| **스킬** | 앱스토어 앱 📱 | Anthropic이나 다른 사람이 만들어 둔 특수 기능 |
| **에이전트** | 부하 직원 🧑‍💼 | Claude가 혼자서 독립적으로 일하는 미니 Claude |

복잡하게 생각하지 마세요. 그냥 **"자주 하는 일을 저장해두는 방법"** 이에요.

## 2. 커맨드 (Command)

### 커맨드가 뭔가요?

커맨드는 **`.md` 파일 하나** 입니다. 그게 전부예요.

Claude Code 채팅창에서 `/커맨드이름` 이라고 치면, Claude가 그 파일 안에 적혀있는 지시사항을 읽고 그대로 실행합니다.

```
사용자: /run-claude-tasks
          ↓
Claude: .claude/commands/run-claude-tasks.md 파일을 읽고 실행!
```

### 파일은 어디에 저장하나요?

두 군데에 저장할 수 있어요:

```
📁 d:\source\specode\
├── 📁 .claude\
│   └── 📁 commands\           ← 이 프로젝트에서만 쓰는 커맨드
│       └── run-claude-tasks.md
│
📁 C:\Users\USER\
└── 📁 .claude\
    └── 📁 commands\           ← 모든 프로젝트에서 쓰는 커맨드 (전역)
```

- **프로젝트 폴더 안** (`.claude/commands/`): 이 프로젝트에서만 사용
- **사용자 홈 폴더** (`C:\Users\이름\.claude\commands\`): 모든 프로젝트에서 사용

### 실제 예시 — `run-claude-tasks.md`

SPECODE에 이미 있는 커맨드예요. 살펴볼게요:

```markdown
# run-claude-tasks — Claude AI 태스크 일괄 처리

## 실행 지시사항

### 1단계: 대기 태스크 조회

curl -s "http://localhost:3000/api/ai/tasks?limit=10" \
  -H "X-API-Key: openclaw-api-key-here"

응답의 data 배열이 비어있으면 종료.

### 2단계: 각 태스크 처리
...
```

이 파일이 있기 때문에, 채팅창에서 그냥 이렇게만 치면 됩니다:

```
/run-claude-tasks
```

그러면 Claude가 알아서 서버에서 대기 태스크를 가져와서, 분석하고, 결과를 등록합니다!

### `$ARGUMENTS` — 커맨드에 값 전달하기

커맨드에 뭔가를 **추가로 전달**하고 싶을 때 `$ARGUMENTS`를 씁니다.

커맨드 파일 안에 이렇게 쓰면:
```markdown
`$ARGUMENTS`가 있으면 URL에 `&taskType=$ARGUMENTS`를 추가한다.
```

채팅에서 이렇게 호출하면:
```
/run-claude-tasks DESIGN
```

`$ARGUMENTS` 자리에 `DESIGN`이 들어가서, `DESIGN` 타입 태스크만 처리합니다.

### 커맨드 만드는 법 — 딱 3단계

**1단계: 파일 만들기**
```
d:\source\specode\.claude\commands\
```
이 폴더에 새 `.md` 파일을 만드세요.

**2단계: 지시사항 쓰기**
```markdown
# 내 커맨드 이름

이렇게 해줘:
1. 첫 번째 할 일
2. 두 번째 할 일
3. 세 번째 할 일
```

**3단계: 채팅에서 쓰기**
```
/내-커맨드-이름
```

끝입니다! 진짜로 이게 전부예요.

## 3. 스킬 (Skill)

### 스킬이 뭔가요?

스킬은 **Anthropic이나 다른 사람이 미리 만들어 둔 커맨드**라고 생각하면 됩니다.
내가 만든 게 아니라 **이미 설치되어 있는** 특수 기능들이에요.

마치 스마트폰에 기본으로 깔려있는 앱처럼요.

### 어떻게 사용하나요?

채팅창에서 `/스킬이름` 으로 바로 씁니다:

```
/sc:git
/simplify
/sc:analyze
```

### SPECODE에서 쓸 수 있는 스킬들

현재 설치된 스킬 중 우리에게 유용한 것들:

| 스킬 이름 | 언제 쓰나요? | 예시 |
|-----------|-------------|------|
| `/sc:git` | 커밋 메시지 자동 작성, git 작업 | 코드 수정 후 `/sc:git` |
| `/simplify` | 내가 짠 코드가 너무 복잡한지 점검 | 리팩토링 후 `/simplify` |
| `/sc:analyze` | 코드 품질, 보안, 성능 분석 | `/sc:analyze src/app/api/` |
| `/sc:implement` | 새 기능 구현 요청 | `/sc:implement 로그인 기능` |
| `/sc:test` | 테스트 코드 생성/실행 | `/sc:test functions/route.ts` |
| `/sc:document` | 문서 자동 생성 | `/sc:document api/functions` |
| `/run-claude-tasks` | AI 태스크 처리 (우리가 만든 것!) | `/run-claude-tasks` |

### 스킬 vs 커맨드 — 차이점

```
커맨드: 내가 .claude/commands/ 에 직접 만든 파일
스킬:   Anthropic이 만들어서 이미 설치된 기능
          → 채팅에서 똑같이 /이름 으로 씀
```

사용하는 방법은 완전히 동일합니다. 만든 사람이 다를 뿐이에요.

### 실전 예시 — `/sc:git`로 커밋하기

```
나: /sc:git
Claude: 변경된 파일들을 확인하고...
        스테이징하고...
        적절한 커밋 메시지를 자동으로 생성합니다:
        "feat: AI 태스크 상세 팝업 추가 및 삭제 기능 구현"
        커밋할까요? (y/n)
```

직접 `git add`, `git commit -m "..."` 타이핑하는 것보다 훨씬 편하죠!

## 4. 에이전트 (Agent)

### 에이전트가 뭔가요?

에이전트는 **"독립적으로 일하는 미니 Claude"** 입니다.

커맨드와 스킬은 Claude가 대화하면서 처리해요. 하지만 에이전트는:

- **혼자서 알아서 일합니다** (내가 계속 보고 있을 필요 없음)
- **여러 일을 동시에** 처리할 수 있습니다
- **복잡한 작업**을 단계별로 나눠서 처리합니다

### 비유로 이해하기

```
커맨드/스킬: 내가 레시피를 읽으면서 직접 요리
에이전트:    전문 요리사를 불러서 "이거 만들어줘"하고 맡기기
              → 요리사가 알아서 장 보고, 재료 다듬고, 요리해서 가져다 줌
```

### 에이전트 파일은 어디에 있나요?

```
📁 d:\source\specode\
└── 📁 .agents\
    └── 📁 workflows\           ← 에이전트 정의 파일들
        └── run-gemini-tasks.md
```

### 실제 예시 — `run-gemini-tasks.md`

SPECODE에 Gemini용 에이전트가 있습니다:

```markdown
---
description: 대기 중인 AI 태스크를 가져와서 분석 후 결과를 SPECODE 서버에 등록합니다.
---
# SPECODE 제미나이(Gemini) 워크로더 실행 가이드

## 1단계: 대기 중인 태스크 조회
http://localhost:3000/api/ai/tasks?limit=5 에 GET 요청

## 2단계: 태스크 처리 루프
모든 태스크에 대해 반복:
- 2-A: 상태를 RUNNING으로 변경
- 2-B: AI가 내용 분석 후 피드백 작성
- 2-C: 결과를 서버에 POST

## 3단계: 요약 응답
```

이걸 Gemini(Google AI)나 Claude에게 "이 워크플로우 실행해줘"라고 하면, 혼자서 1~3단계를 다 처리합니다.

### 에이전트의 종류 (Claude Code 내장)

Claude Code 안에는 이미 여러 종류의 에이전트가 내장되어 있어요:

| 에이전트 | 특기 |
|---------|------|
| **general-purpose** | 뭐든지 다 함 (기본값) |
| **Explore** | 코드베이스 탐색 전문가 |
| **Plan** | 계획 수립 전문가 |
| **claude-code-guide** | Claude Code 사용법 안내 전문가 |

Claude가 복잡한 작업을 할 때 이 에이전트들을 알아서 골라서 씁니다. 우리가 직접 뭘 고를 필요는 없어요.

### 에이전트 만드는 법

`.agents/workflows/` 폴더에 `.md` 파일을 만들면 됩니다.

파일 맨 위에 이렇게 설명을 씁니다:

```markdown
---
description: 이 에이전트가 하는 일을 한 줄로 설명
---

# 에이전트 이름

## 해야 할 일 1단계
...

## 해야 할 일 2단계
...
```

## 5. 세 가지 비교

### 언제 뭘 써야 하나?

```
┌─────────────────────────────────────────────────────────────┐
│                         내가 원하는 것                         │
├─────────────────┬───────────────────────────────────────────┤
│ 자주 하는 작업을  │ → 커맨드 만들기                             │
│ 빠르게 실행하고 싶다│   .claude/commands/내-커맨드.md             │
├─────────────────┼───────────────────────────────────────────┤
│ 남이 만든 좋은   │ → 스킬 사용하기                              │
│ 기능을 쓰고 싶다  │   /sc:git, /simplify, /sc:analyze           │
├─────────────────┼───────────────────────────────────────────┤
│ 복잡한 작업을    │ → 에이전트 만들기                            │
│ 자동으로 처리하고 싶다│  .agents/workflows/내-에이전트.md        │
└─────────────────┴───────────────────────────────────────────┘
```

### 핵심 차이점 정리

| 구분 | 커맨드 | 스킬 | 에이전트 |
|------|--------|------|---------|
| 누가 만드나 | 나 | Anthropic/다른 사람 | 나 |
| 어디에 저장 | `.claude/commands/` | 자동 설치됨 | `.agents/workflows/` |
| 어떻게 씀 | `/커맨드이름` | `/스킬이름` | 워크플로우로 실행 |
| 복잡도 | 간단한 지시사항 | 복잡한 기능 포함 | 여러 단계 자동화 |
| 독립 실행 | ❌ 대화 안에서 | ❌ 대화 안에서 | ✅ 혼자서 실행 |

## 6. 나만의 커맨드 만들기

### 실습: "DB 체크" 커맨드 만들기

SPECODE의 데이터베이스 상태를 확인하는 커맨드를 만들어봅시다.

**Step 1: 파일 만들기**

`d:\source\specode\.claude\commands\db-check.md` 파일을 새로 만드세요.

**Step 2: 내용 쓰기**

```markdown
# db-check — 데이터베이스 상태 확인

SPECODE 프로젝트의 데이터베이스 상태를 확인한다.

## 실행 순서

### 1단계: Prisma 상태 확인
다음 명령어를 실행한다:
```bash
cd d:/source/specode && npx prisma db pull --print 2>&1 | head -20
```

### 2단계: 주요 테이블 레코드 수 확인
SQLite 데이터베이스에서 각 테이블의 레코드 수를 확인한다:
```bash
cd d:/source/specode && npx prisma studio --browser none &
```

### 3단계: 결과 요약
확인된 내용을 표 형식으로 정리해서 보여준다.
```

**Step 3: 채팅에서 사용**

```
/db-check
```

Claude가 위에 적힌 순서대로 실행합니다!

### 커맨드 작성 팁

```markdown
# 좋은 커맨드 파일 구조

# 커맨드 이름 — 한 줄 설명

이 커맨드가 언제, 왜 사용되는지 1-2줄로 설명.

## 단계별 지시사항

### 1단계: 무엇을 하는가
구체적으로 무엇을 해야 하는지 씁니다.

### 2단계: 결과 처리
결과를 어떻게 처리할지 씁니다.

## 주의사항
- 조심해야 할 것이 있으면 여기에
- 실패하면 어떻게 해야 하는지
```

**팁 목록:**

✅ **구체적으로 써라** — "API 호출해라"보다 "GET http://... 이렇게 호출해라"가 좋음
✅ **단계를 나눠라** — 한 번에 많은 걸 하지 말고, 1단계·2단계·3단계로
✅ **실패 처리를 써라** — "오류 나면 FAILED로 전송해라" 같은 내용 포함
✅ **한국어로 써도 됨** — Claude는 한국어 지시사항도 완벽히 이해함

## 7. 나만의 에이전트 만들기

### 실습: "AI 태스크 리포트" 에이전트 만들기

매일 AI 태스크 처리 결과를 요약하는 에이전트를 만들어봅시다.

**Step 1: 파일 만들기**

`d:\source\specode\.agents\workflows\daily-report.md` 파일을 새로 만드세요.

**Step 2: 내용 쓰기**

```markdown
---
description: SPECODE AI 태스크 처리 현황을 일별로 요약하여 보고합니다.
---

# SPECODE 일일 AI 태스크 리포트

오늘 처리된 AI 태스크 현황을 수집하고 요약 리포트를 작성합니다.

## 1단계: 데이터 수집

다음 API를 호출하여 오늘의 태스크 현황을 가져옵니다:

```bash
curl -s "http://localhost:3000/api/ai-tasks?page=1&pageSize=50"
```

## 2단계: 통계 계산

가져온 데이터에서 다음을 계산합니다:
- 전체 태스크 수
- 상태별 집계 (SUCCESS, FAILED, RUNNING, NONE)
- 작업 유형별 집계 (DESIGN, INSPECT, IMPLEMENT)
- 평균 처리 시간

## 3단계: 리포트 작성

다음 형식으로 마크다운 리포트를 작성합니다:

```markdown
# AI 태스크 일일 리포트 — {날짜}

## 요약
- 전체 처리: N건
- 성공률: N%

## 상태별 현황
| 상태 | 건수 |
|------|------|
| 완료 | N |
| 실패 | N |
...

## 주요 완료 태스크 목록
1. ...
2. ...

## 개선이 필요한 실패 태스크
1. ...
```

## 4단계: 파일 저장 (선택)

리포트를 `d:/source/specode/reports/{날짜}-ai-report.md` 로 저장합니다.
```

**Step 3: 실행하기**

채팅창에서:
```
오늘의 AI 태스크 리포트 만들어줘. .agents/workflows/daily-report.md 워크플로우로 실행해줘.
```

### 에이전트 vs 커맨드 선택 기준

```
이런 경우 → 커맨드 (간단!)
  - 딱 한 가지 일을 함
  - API 한두 번 호출하고 끝
  - 결과를 화면에 보여주기만 함

이런 경우 → 에이전트 (복잡!)
  - 여러 단계가 있고, 각 단계 결과에 따라 다음 행동이 달라짐
  - 파일 읽고, 분석하고, 결과 저장하는 등 복합 작업
  - 시간이 오래 걸리는 작업 (파일 100개 분석 등)
```

## 8. SPECODE에서 실제로 쓰는 패턴

### 패턴 1: AI 태스크 처리 (현재 사용 중)

```
채팅: /run-claude-tasks
       ↓
커맨드 파일: .claude/commands/run-claude-tasks.md 실행
       ↓
Claude가 순서대로:
  1. GET /api/ai/tasks → 대기 태스크 가져오기
  2. PATCH /api/ai/tasks/{id}/start → 시작 표시
  3. spec 내용 분석 (INSPECT/DESIGN/IMPLEMENT)
  4. Python으로 결과 POST (한글 깨짐 방지)
```

### 패턴 2: 코드 수정 후 바로 커밋

```
(코드 수정 완료)
채팅: /sc:git
       ↓
스킬이 알아서:
  1. 변경 파일 확인 (git status)
  2. 변경 내용 파악 (git diff)
  3. 적절한 커밋 메시지 자동 생성
  4. git add + git commit
```

### 패턴 3: 새 기능 구현 + 리뷰 콤보

```

       ↓
구현 완료 후
채팅: /simplify
       ↓
스킬이 코드 품질 점검:
  - 중복 코드 없나?
  - 더 단순하게 쓸 수 있나?
  - 보안 문제 없나?
```

### 패턴 4: 작업 타입별 태스크 처리

$ARGUMENTS = "DESIGN"
