# SPECODE — 시스템 소개

> 이 문서 하나로 SPECODE가 무엇인지, 왜 만들었는지, 어떻게 동작하는지 이해할 수 있습니다.

---

## SPECODE는 이것이다

**SI(시스템통합) 프로젝트를 만드는 사람들이, AI와 함께 설계하고 개발하는 플랫폼.**

SI 프로젝트에서 가장 힘든 일은 코딩이 아닙니다.
RFP를 읽고 요구사항을 정리하고, 화면을 설계하고, 기능을 명세하고,
개발자가 구현할 수 있을 만큼 구체적으로 만드는 것 — 이 과정에서 생산성이 가장 많이 사라집니다.

SPECODE는 이 문제를 해결합니다.

기획자(PM)가 요구사항을 등록하면, AI가 검토하고 피드백을 줍니다.
설계자가 화면·영역·기능을 정의하면, AI가 상세 설계를 작성합니다.
개발자가 기능 명세를 확인하면, AI가 구현 가이드와 코드를 제안합니다.

---

## 왜 만들었는가

### SI 프로젝트의 현실

SI 프로젝트는 고질적인 문제가 있습니다.

- **요구사항이 모호하다**: RFP 문장 하나가 개발팀에 전달될 때쯤이면 원래 의도가 희석되거나 변형되어 있습니다.
- **명세가 부족하다**: 기능을 등록하지만 "어떻게 동작해야 하는가"에 대한 설명이 빈약합니다. 개발자가 스스로 해석하다 보면 방향이 달라집니다.
- **추적이 안 된다**: RFP의 어느 요구사항이 어느 화면으로 구현됐는지, 어느 기능이 어떤 상태인지 한눈에 보기 어렵습니다.
- **설계 비용이 크다**: 상세 설계 문서를 작성하는 데 많은 시간이 걸립니다. 결국 대충 넘어가거나 생략됩니다.

### SPECODE의 답

계층 구조로 모든 것을 연결하고, AI가 각 단계를 보조하게 합니다.

```
과업(RFP 항목)
  └─ 요구사항
        ├─ 사용자 스토리
        └─ 단위업무
              └─ 화면
                    └─ 영역
                          └─ 기능  ← AI가 가장 많이 일하는 곳
```

과업에서 기능까지 단방향으로 연결되어 있어서,
기능 하나를 보면 그게 어느 요구사항에서 나왔는지,
어느 RFP 항목을 구현하는 것인지 바로 추적할 수 있습니다.

## 도메인 개념 설명

### 과업 (Task)
RFP의 대항목입니다. 예: "회원관리 기능 개발", "결제 시스템 구축".
과업 하나가 여러 요구사항을 담습니다.

### 요구사항 (Requirement)
"시스템은 ~해야 한다"는 단위입니다.
원문(계약서/RFP 근거)과 현행화된 내용(협의 반영)을 구분해서 관리합니다.
설계가 변경되어도 원래 계약 근거는 보존됩니다.

### 사용자 스토리 (UserStory)
요구사항을 사용자 관점으로 풀어쓴 것입니다.
"누가, 무엇을 하면, 어떤 가치를 얻는가" + 인수기준(완료 판단 기준)으로 구성됩니다.
화면과 매핑할 수 있어서 어느 화면이 어느 스토리를 구현하는지 연결됩니다.

### 단위업무 (UnitWork)
화면들의 묶음입니다. 예: "주문 처리" 단위업무 아래 주문목록 화면, 주문상세 화면, 주문취소 화면이 묶입니다.
요구사항을 실제 개발 단위로 쪼개는 중간 계층입니다.

### 화면 (Screen)
사용자가 실제로 보는 UI 페이지입니다. LIST, DETAIL, POPUP, TAB 등 유형을 분류합니다.
대분류·중분류·소분류로 메뉴 체계를 구성할 수 있습니다.

### 영역 (Area)
화면 내의 논리적 구획입니다. 예: 검색 조건 영역, 결과 목록 영역, 상세 폼 영역.
영역 유형은 SEARCH / GRID / FORM / INFO_CARD / TAB / FULL_SCREEN으로 분류합니다.
Excalidraw로 도안을 그리거나, AI에게 설계를 요청할 수 있습니다.

### 기능 (Function)
SPECODE에서 가장 중요한 단위입니다. 전체 시스템 상호작용의 80%가 기능 단위로 발생합니다.
사용자가 수행하는 구체적인 동작 하나입니다. 예: "검색 실행", "저장", "엑셀 다운로드".
기능마다 명세(spec)를 작성하고, AI에게 검토·상세설계·구현 가이드를 요청합니다.

### 기능의 상태 흐름
기능은 아래 순서로 진행됩니다:
```
DRAFT  →  검토 요청(REVIEW_REQ)  →  AI 검토 완료(REVIEW_DONE)
       →  설계 요청(DESIGN_REQ)  →  AI 설계 완료(DESIGN_DONE)
       →  컨펌(CONFIRM_Y)
       →  구현 요청(IMPL_REQ)    →  AI 구현 완료(IMPL_DONE)
```
상태가 바뀔 때마다 AI 태스크가 자동으로 생성됩니다.

## 주요 기능 설명

### AI 자동화 파이프라인

SPECODE의 핵심입니다. 기능의 상태가 바뀌면 AI 태스크(큐)가 생성되고,
AI 워커(`/run-claude-tasks`)가 이를 처리해서 결과를 저장합니다.

- **검토(INSPECT)**: 기능 명세가 충분히 구체적인가? 빠진 게 없는가? AI가 피드백을 작성합니다.
- **상세 설계(DESIGN)**: 기능 명세를 보고 데이터 흐름, API 설계, DB 접근, 예외 처리를 포함한 상세 설계 문서를 생성합니다. 이미지 첨부 시 화면 레이아웃도 반영합니다.
- **구현 가이드(IMPLEMENT)**: 설계를 바탕으로 실제 구현 가능한 코드와 파일 경로를 제안합니다.
- **목업 생성(MOCKUP)**: 화면 또는 영역 단위로 HTML 목업을 자동 생성합니다. 생성 직후 iframe으로 바로 확인할 수 있습니다.

모든 AI 처리는 비동기입니다. 요청만 해두면 백그라운드에서 처리되고, 완료되면 결과가 자동으로 저장됩니다.

### 기획 보드 (PlanningBoard)

요구사항들을 묶어서 IA(정보구조), 프로세스 흐름도, HTML 목업을 AI가 생성하는 공간입니다.
단순히 기능 하나가 아니라 여러 요구사항을 참고해서 큰 그림을 그릴 때 사용합니다.

- **IA**: 메뉴 체계, 정보 구조도를 마크다운으로 생성
- **PROCESS**: 업무 흐름도를 Mermaid 다이어그램으로 생성
- **MOCKUP**: 실제 렌더링 가능한 HTML 목업 생성

생성된 기획은 "화면으로 승격"해서 실제 화면 설계로 연결할 수 있습니다.

### 표준 가이드 (StandardGuide)

개발 표준을 관리하는 공간입니다. UI, API, 보안, DB, 파일, 인증, 에러 처리 등 카테고리별로 가이드 문서를 작성하고, AI가 검토합니다.
AI가 기능 설계를 할 때 이 표준 가이드를 참조해서 일관성 있는 결과를 만듭니다.

### PRD 내보내기

설계가 완료된 단위업무·화면·영역·기능을 마크다운 PRD 문서로 내보낼 수 있습니다.
마지막 다운로드 이후 변경된 내용이 자동으로 상단에 표시됩니다(Diff 기반 변경 노트).
개발자에게 명세서를 전달할 때 사용합니다.

### AI 설계 가져오기 (AI Import)

개발자들이 Claude 프로젝트에서 직접 설계하고, 그 결과를 JSON으로 SPECODE에 빠르게 등록하는 기능입니다.
두 가지 방향으로 동작합니다:

1. **단위업무/화면/영역/기능 가져오기**: Claude와 대화로 화면 설계 → JSON → SPECODE 등록
2. **과업/요구사항/사용자스토리 가져오기**: Claude와 대화로 요구사항 분석 → JSON → SPECODE 등록

Claude 프로젝트에 시스템 프롬프트를 넣으면, Claude가 설계 파트너로서 4단계(또는 3단계) 프로세스를 안내합니다. systemId가 있는 항목은 수정, 없는 항목은 신규 등록됩니다.

### 일괄 설계 (BulkDesign)

화면→영역→기능 계층을 탐색하면서 명세를 한 곳에서 편집하는 화면입니다.
여러 기능의 명세를 빠르게 작성하거나, 연결된 요구사항 원문을 참고하면서 작업할 수 있습니다.
선택 범위 기준으로 AI 설계 태스크를 일괄 생성할 수도 있습니다.

### 변경 이력 (ContentVersion)

요구사항, 기능 명세 등 중요한 텍스트 필드가 변경될 때마다 이전 내용을 저장합니다.
언제든 Diff 뷰어로 이전/현재 내용을 비교할 수 있어서 설계 변경 이력을 추적할 수 있습니다.

### MCP 서버

SPECODE 데이터에 외부 AI 에이전트가 직접 접근할 수 있는 인터페이스입니다.
Claude Code(로컬), claude.ai, 다른 AI 도구가 SPECODE의 화면·영역·기능 데이터를 읽고, AI 작업을 요청하고, 결과를 저장할 수 있습니다.
개발자가 자신의 AI 도구에서 바로 SPECODE와 대화하는 것이 목표입니다.

### DB 스키마 관리

개발 중인 시스템의 테이블 DDL을 SPECODE에서 관리합니다.
컬럼 메타데이터 파싱, 관계도 시각화를 지원합니다. AI가 기능 설계 시 실제 DB 구조를 참고할 수 있습니다.

### 요구사항 허브 (ReqHub)

요구사항 하나를 중심으로 연결된 단위업무·화면·기능 전체를 한눈에 보는 통합 뷰입니다.
"이 요구사항이 실제로 어디서 구현되고 있는가"를 확인할 때 사용합니다.

## 대시보드로 보는 현재 상태

시스템 진입 시 보이는 대시보드는 두 가지입니다:

- **대시보드**: 기능 상태별 건수(DRAFT/검토/설계/컨펌/구현 중/구현완료), 오늘의 AI 처리 건수, 최근 AI 활동 피드
- **개발 현황판**: 단계별 진행률(설계율, 구현율), 기능별 상태 테이블

언제든 현재 프로젝트가 어느 단계에 있는지, 무엇이 막혀 있는지 파악할 수 있습니다.

## AI 워커 동작 방식

SPECODE의 AI 처리는 폴링 기반 비동기 큐입니다.

```
1. 기능 상태 변경 → AI 태스크(큐)가 자동 생성됨 (상태: NONE)
2. AI 워커가 주기적으로 대기 태스크를 조회
3. 태스크 하나씩 처리: 시작(RUNNING) → 분석 → 완료(SUCCESS)
4. 완료 훅이 실행: AI 결과를 해당 기능/영역/화면에 자동 저장
5. 상태 자동 업데이트 (예: DESIGN_REQ → DESIGN_DONE)
```

현재 AI 워커는 Claude Code의 `/run-claude-tasks` 슬래시 커맨드로 수동 실행합니다.
각 태스크 유형별 분석 지침은 `docs/prompts/` 폴더의 마크다운 파일로 관리됩니다.

## 현재 상태와 앞으로의 계획

### 현재 (단일 테넌트)

지금은 한 팀이 사용하는 단일 프로젝트 구조입니다.
로그인·권한 관리 없이 하나의 데이터 공간을 공유합니다.

### 목표 (멀티 테넌트 SaaS)

SPECODE를 여러 팀이 독립적으로 사용할 수 있는 SaaS 플랫폼으로 확장합니다.

**추가될 개념:**
- **프로젝트**: 모든 데이터의 최상위 컨테이너. 팀마다 독립된 데이터 공간
- **사용자 & 로그인**: 이메일/소셜 로그인(Google, GitHub)
- **역할(Role)**: OWNER / ADMIN / PM / DESIGNER / DEVELOPER / VIEWER — 역할별 접근 권한 분리
- **구독 플랜**: FREE (1프로젝트/3명/월30건) → STARTER → PRO → ENTERPRISE

**확장 로드맵:**
1. **P0 (즉시)**: JWT 인증 + 프로젝트 개념 도입
2. **P1 (1개월)**: 멤버 초대 + 역할 권한 + 회원가입/로그인 UI
3. **P2 (3개월)**: AI 사용량 추적 + 구독/결제 연동
4. **P3 (6개월)**: ENTERPRISE 기능 (SSO, 전용 AI 키, 감사 로그)

## 한 줄 요약

> SPECODE는 SI 프로젝트의 기획·설계·개발 과정에서 생기는 모든 산출물을 계층적으로 연결하고, 각 단계마다 AI가 실질적인 도움을 주는 협업 플랫폼이다.

인간이 방향을 잡고, AI가 구체화한다. 그 결과물이 항상 추적 가능하고, 다음 단계로 이어진다.
