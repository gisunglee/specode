## 핵심 테이블 관계

```
tb_project
  └─ tb_task          (projectId FK)
       └─ tb_requirement   (taskId FK)
            ├─ tb_user_story    (requirementId FK)
            │    └─ tb_acceptance_criteria
            └─ tb_unit_work     (requirementId FK)  ← 요구사항과 N:1
                 └─ tb_screen   (unitWorkId FK)
                      └─ tb_area    (screenId FK)
                           └─ tb_function  (areaId FK)
                                └─ tb_ai_task  (refPkId FK, refTableName='tb_function')

tb_project_member  (projectId + userId 복합 UK)
tb_planning_draft  (projectId FK)
tb_standard_guide  (projectId FK, categoryId FK nullable=미분류)
tb_sequence        (prefix PK, lastNum)
tb_content_version (tableName + refPkId, 이력 저장)
```

AI 태스크 spec 구성, 브레드크럼 표시 등에 필요.
단순 기능 목록이라도 area → screen → unitWork 포함 권장.

```ts
// 올바른 기능 조회 패턴
{
  where: { areaId, delYn: 'N' },
  include: {
    area: {
      include: {
        screen: {
          include: { unitWork: true }
        }
      }
    }
  },
  orderBy: { sortOrder: 'asc' }
}
```

## AI 태스크 spec 구성 패턴

기능의 AI 태스크 생성 시 이 구조로 spec JSON 구성.
상위 계층 빠지면 AI 결과 품질 크게 저하.

```ts
const spec = {
  function: {
    systemId, name, spec, status, functionType
  },
  area: {
    name, areaType, description
  },
  screen: {
    name, layoutData
  },
  unitWork: {
    name, description
  },
  requirement: {
    name, originalContent, currentContent, detailSpec
  },
  project: {
    name
  }
}
```

기획 보드(PLANNING) spec 구성:
```ts
const spec = {
  planType,          // IA | PROCESS | MOCKUP | ERD
  manualInfo,        // 수동 작성 내용
  comment,           // 사용자 추가 요청
  requirements: [    // 선택된 요구사항 목록
    { systemId, name, detailSpec, discussionMd, content }
  ]
}
```

## systemId 채번 트랜잭션

동시 요청 시 중복 발생 방지. 반드시 트랜잭션으로 처리.

```ts
async function generateSystemId(prefix: string): Promise<string> {
  return await prisma.$transaction(async (tx) => {
    const seq = await tx.sequence.update({
      where: { prefix },
      data: { lastNum: { increment: 1 } },
    });
    return `${prefix}-${String(seq.lastNum).padStart(5, '0')}`;
  });
}
```

## 콘텐츠 이력 저장 패턴

명세 필드 저장 시 이전 값 이력 적재.
`saveContentVersion()` 유틸 사용.

```ts
// 저장 전 이전 값 이력 적재
await saveContentVersion({
  tableName: 'tb_function',
  refPkId: functionId,
  fieldName: 'spec',
  content: existingFunction.spec,  // 이전 값
  changeType: 'MANUAL',
  changedBy: userId,
});

// 이후 업데이트
await prisma.function.update({ ... });
```

## 소프트 삭제 cascade 패턴

DB cascade 없음. 코드에서 직접 처리.

```ts
// 화면 삭제 예시
await prisma.$transaction([
  // 기능 먼저
  prisma.function.updateMany({
    where: { area: { screenId }, delYn: 'N' },
    data: { delYn: 'Y' }
  }),
  // 영역
  prisma.area.updateMany({
    where: { screenId, delYn: 'N' },
    data: { delYn: 'Y' }
  }),
  // 화면
  prisma.screen.update({
    where: { screenId },
    data: { delYn: 'Y' }
  }),
]);
```

## 정렬 순서 신규 생성 패턴

```ts
const maxOrder = await prisma.requirement.aggregate({
  where: { taskId, delYn: 'N' },
  _max: { sortOrder: true }
});
const sortOrder = (maxOrder._max.sortOrder ?? 0) + 1;
```

## 주요 인덱스 권장

성능에 직접 영향. 마이그레이션 시 반드시 추가.

```sql
-- AI 태스크 폴링 (가장 빈번한 쿼리)
CREATE INDEX idx_ai_task_status ON tb_ai_task(task_status, created_at);

-- 프로젝트별 기능 조회
CREATE INDEX idx_function_area ON tb_function(area_id, del_yn, sort_order);

-- 멤버 조회
CREATE UNIQUE INDEX idx_project_member ON tb_project_member(project_id, user_id);
```
