Why SPECODE

SI 프로젝트의 고질적인 문제들

코딩이 아닌 기획과 설계 단계에서 생산성이 가장 많이 사라집니다.

🌫️
요구사항이 모호하다
RFP 문장 하나가 개발팀에 전달될 때쯤이면 원래 의도가 희석되거나 변형되어 있습니다.
📭
명세가 부족하다
기능을 등록하지만 "어떻게 동작해야 하는가"에 대한 설명이 빈약합니다. 개발자가 스스로 해석하다 보면 방향이 달라집니다.
🔍
추적이 안 된다
RFP의 어느 요구사항이 어느 화면으로 구현됐는지, 어느 기능이 어떤 상태인지 한눈에 보기 어렵습니다.
⏰
설계 비용이 크다
상세 설계 문서를 작성하는 데 많은 시간이 걸립니다. 결국 대충 넘어가거나 생략됩니다.
SPECODE는 계층 구조로 모든 것을 연결하고, AI가 각 단계를 보조하게 합니다.
기능 하나를 보면 그게 어느 요구사항에서 나왔는지, 어느 RFP 항목을 구현하는 것인지 바로 추적할 수 있습니다.

Domain Structure

도메인 계층 구조

과업에서 기능까지 단방향으로 연결된 6계층 구조. 모든 설계 산출물이 하나의 트리 안에 있습니다.

계층 트리
📋 과업 (Task)
  └─ 📌 요구사항 (Requirement)
       ├─ 👤 사용자 스토리 (UserStory)
       └─ 🔷 단위업무 (UnitWork)
              └─ 🖥️ 화면 (Screen)
                     └─ ▦ 영역 (Area)
                            └─ ⚙️ 기능 (Function)AI
과업 Task
RFP의 대항목. "회원관리 기능 개발" 등 과업 하나가 여러 요구사항을 담습니다.
요구사항 Requirement
"시스템은 ~해야 한다"는 단위. 원문(계약서/RFP)과 현행화된 내용을 구분 관리합니다. 원래 계약 근거는 항상 보존됩니다.
단위업무 UnitWork
화면들의 묶음. "주문 처리" 단위업무 아래 주문목록·상세·취소 화면이 묶입니다. 요구사항을 실제 개발 단위로 쪼개는 중간 계층.
화면 Screen
사용자가 실제로 보는 UI 페이지. LIST / DETAIL / POPUP / TAB 유형. 대·중·소분류로 메뉴 체계 구성.
영역 Area
화면 내의 논리적 구획. SEARCH / GRID / FORM / INFO_CARD / TAB / FULL_SCREEN 유형. Excalidraw 도안 지원.
기능 ⭐ Function
SPECODE에서 가장 중요한 단위. 전체 시스템 상호작용의 80%가 기능 단위로 발생. AI가 검토·설계·구현 가이드를 생성합니다.
⚙️ 기능(Function) 상태 흐름
DRAFT → 검토 요청 → AI 처리 중 → 검토 완료 → 설계 요청 → AI 처리 중 → 설계 완료 → 컨펌 → 구현 완료

Features

주요 기능

AI 자동화 파이프라인

AI Pipeline

AI 워커 동작 방식

폴링 기반 비동기 큐로 AI 태스크를 처리합니다. 요청만 해두면 백그라운드에서 처리되고 결과가 자동으로 저장됩니다.

STEP 01
🔔
태스크 생성
기능 상태 변경 시 AI 태스크가 자동 생성됩니다.
NONE
STEP 02
🔍
폴링 조회
AI 워커가 주기적으로 대기 태스크를 조회합니다.
GET /api/ai/tasks
STEP 03
⚡
AI 분석·생성
태스크를 시작하고 spec 분석, 이미지 분석(첨부 시), 설계 문서를 생성합니다.
RUNNING
STEP 04
✅
완료 훅 실행
AI 결과를 해당 기능·영역·화면에 자동 저장하고 상태를 업데이트합니다.
SUCCESS
# 현재 AI 워커: Claude Code 슬래시 커맨드 (수동 실행) $ /run-claude-tasks 1. 기능 상태 변경 → AI 태스크 자동 생성 (NONE) 2. 워커가 대기 태스크 조회 → 시작 (RUNNING) 3. spec 분석 + 이미지 분석 → 마크다운 결과 생성 4. 결과 저장 → SUCCESS 5. 완료 훅: 기능 상태 자동 업데이트 → DESIGN_DONE # 태스크 유형별 분석 지침: docs/prompts/{table}-{type}.md
